# Написание навыков

Хорошо написанный навык — это половина успеха графа. Плохой контент делает граф бесполезным, отличный — мощным инструментом.

## Принципы написания

### 1. Одна идея = один навык

**Плохо:**
```markdown
# Основные принципы

Принцип 1: X
Принцип 2: Y
Принцип 3: Z
```

**Хорошо:**
```markdown
# Принцип 1: X

[Полное объяснение принципа]

# Принцип 2: Y

[Полное объяснение принципа]
```

**Правило:** Каждый навык фокусируется на одной законченной мысли.

### 2. Конкретные примеры

**Плохо (абстрактно):**
```markdown
Всегда пишите понятный код.
```

**Хорошо (с примерами):**
```python
# Плохо
def d(x):
    return x * 86400

# Хорошо
def days_to_seconds(days):
    return x * 86400
```

**Правило:** Код/примеры лучше 1000 слов объяснения.

### 3. Плохие vs хорошие примеры

Всегда показывайте что не делать и что делать:

**Плохо:**
```markdown
Используйте осмысленные имена.
```

**Хорошо:**
```markdown
**Плохо:**
```python
def d(x):
    return x * 86400
```

**Хорошо:**
```python
def days_to_seconds(days):
    return x * 86400
```

**Почему:** [Объяснение почему плохой подход.]

## Лучшие практики

### Избегайте лекций

**Плохо:**
```markdown
Важно писать хороший код. Хороший код имеет понятные имена. Он также обрабатыват ошибки корректно. Также важны тесты...
```

**Хорошо:**
```markdown
## Principles

1. **Используйте осмысленные имена** — Они раскрывают намерение
2. **Обрабатывайте ошибки** — Не глотайте исключения
3. **Пишите тесты** — Они ловят регрессии

## Examples

[Конкретные примеры]
```

**Правило:** Структура > поток текста.

### Пишите активно, не пассивно

**Плохо (пассивный):**
```markdown
Код должен быть написан правильно.
Тесты должны быть созданы.
```

**Хорошо (активный):**
```markdown
Пишите код правильно.
Создавайте тесты.
```

**Правило:** Используйте повелительное наклонение (делай X, не X должно быть сделано).

### Избегайте "мы" и "вы"

**Плохо:**
```markdown
Вы должны писать хороший код.
Мы рекомендуем тестирование.
```

**Хорошо:**
```markdown
Пишите хороший код.
Рекомендуется тестирование.
```

**Правило:** Обращайтесь напрямую к читателю.

## Длина и глубина

### Правило 80/20

- **80% базовые концепции:** Фундаментальные принципы
- **20% продвинутые детали:** Специфические техники

**Почему:** Большинство пользователей остановится на базовом уровне.

### Правило читаемости

- **Короткие параграфы:** 3-4 строки максимум
- **Короткие предложения:** 15-20 слов
- **Используйте списки:** Для множества элементов
- **Используйте заголовки:** Для разделения идей

## Код и примеры

### Код должен быть копируемым

**Плохо (фрагменты):**
```python
# ...и вот тут часть кода
def function():
    ...
```

**Хорошо (полный):**
```python
def calculate_total(items):
    total = 0
    for item in items:
        total += item.price
    return total
```

**Правило:** Код должен работать если скопировать.

### Объясняйте "почему", не только "что"

**Плохо:**
```python
def calculate_total(items):
    total = 0  # Инициализируем
    for item in items:  # Цикл
        total += item.price  # Добавляем цену
    return total  # Возвращаем
```

**Хорошо:**
```python
def calculate_total(items):
    """
    Считает общую сумму заказа.

    Используем аккумулировать вместо sum() для ясности
    и чтобы показать явно цикл для обучения.
    """
    total = 0
    for item in items:
        total += item.price
    return total
```

**Правило:** Комментарии объясняют почему, не что.

## Частые ошибки

### 1. Слишком абстрактно

**Симптом:** Много слов, мало примеров
**Решение:** Добавьте конкретные примеры

### 2. Слишком длинно

**Симптом:** Навык > 2000 слов
**Решение:** Разбейте на 2-3 навыка

### 3. Нет контекста

**Симптом:** "Используйте X" без объяснения когда/почему
**Решение:** Добавьте контекст применения

### 4. Только теория

**Симптом:** Много концепций, нет практики
**Решение:** Добавьте "как применять" секцию
