# Код-ревью (Code Review)

## Overview

Код-ревью — это процесс систематического просмотра кода другими разработчиками для нахождения ошибок, улучшения качества и обмена знаниями. Хорошее код-ревью повышает качество кода, обучает команду и создаёт культуру качества.

## Ключевые принципы

### 1. Code Review — это обучение, а не критика

**Плохо:**
```markdown
Этот код ужасен. Ты почему так пишешь?
Переделывай.
```

**Хорошо:**
```markdown
Предлагаю разбить эту функцию на две части:
- `calculate_total()`
- `calculate_discount()`

Это улучшит читаемость и тестируемость.
```

### 2. Конструктивная критика с конкретными предложениями

**Плохо:**
```markdown
Не используй магические числа.
```

**Хорошо:**
```markdown
Хорошая идея использовать константы! Предлагаю:

```python
DISCOUNT_RATE = 0.1  # 10% скидка

def apply_discount(price, discount_rate=DISCOUNT_RATE):
    return price * discount_rate
```

Это делает код более читаемым и легко изменяемым.
```

### 3. Сосредоточься на важных аспектах

**Что проверять:**

| Аспект | Что проверять | Пример |
|---------|---------------|---------|
| Корректность | Логика кода | Правильны ли условия? |
| Производительность | Алгоритмическая эффективность | Можно ли улучшить? |
| Читаемость | Понятность кода | Читается ли легко? |
| Тестируемость | Покрытость тестами | Есть ли тесты? |
| Безопасность | Уязвимости | Есть ли проблемы безопасности? |
| Документация | Наличие комментариев | Есть ли docstrings? |
| Стиль кода | Следование стандартам | PEP 8, соглашения команды |

### 4. Предоставьте контекст

**Пример:**
```markdown
Я заметил, что функция `calculate_total()` не обрабатывает отрицательные цены. Это может привести к проблемам с расчётом скидок.

**Проблема:** Если цена отрицательная (возможно при возврате товара), расчёт будет неправильным.

**Рекомендация:** Добавь валидацию входных данных в начале функции:

```python
def calculate_total(items):
    """Calculate total of item prices.
    
    Args:
        items: List of item dictionaries with 'price' key.
    
    Returns:
        Total price, ignoring negative or zero prices.
    """
    total = 0
    for item in items:
        price = item.get('price', 0)
        if price > 0:
            total += price
    return total
```

**Преимущества:**
- Код-ревьюер понимает контекст проблемы
- Предложение с объяснением почему
- Код становится более надёжным

### 5. Пишите понятные и конкретные комментарии

**Плохо:**
```python
# Убери это
```

**Хорошо:**
```python
# Предлагаю использовать list comprehension для более чистого кода:
# Было:
total = 0
for item in items:
    total += item.price

# Стало:
total = sum(item['price'] for item in items if item.get('price', 0) > 0)

Это более Pythonic и читаемо.
```

**Уровни комментариев:**

**1. Объяснение "почему":**
```markdown
```python
# Хочу понять, почему мы используем цикл for вместо comprehension.
# Цикл for более понятен для сложных операций
# Я добавлю docstring для объяснения выбора.

items = []
for item in database_items:
    items.append(process_item(item))
```

**2. Альтернативное решение:**
```markdown
```python
# Альтернатива: использование generator expression
# Преимущество: не создаёт весь список в памяти
items = (process_item(item) for item in database_items)

# Или использование itertools для очень больших данных
import itertools
for chunk in itertools.chunks(huge_list, 1000):
    process_chunk(chunk)
```

**3. Ссылки на документацию:**
```markdown
```python
# Если функция использует внешнюю библиотеку, ссылаемся на документацию
from zvec import Collection  # ← это нормально для open-source библиотек

# Если нестандартное поведение, объясняем почему
# PEP 8 не рекомендует использовать mutable default arguments
# Но если есть веская причина, добавляем объясняющий комментарий
def process_data(data, use_cache=True):  # ← параметр с понятным именем
    """
    Process data with optional caching.
    
    Args:
        data: Input data to process.
        use_cache: If True, use cached results (default).
    
    Returns:
        Processed data.
    """
```

### 6. Будьте позитивным и поддерживайте культуру обучения

**Пример:**
```markdown
Отличная работа над этой функцией! Я вижу, что ты проделал большую работу над оптимизацией производительности. Предлагаю создать отдельную функцию `calculate_total_fast()` для кеширования результатов — это поможет при работе с большими объёмами данных.
```

**Преимущества:**
- Создаётся позитивная атмосфера
- Развивается культура обучения
- Мотивирует команду

### 7. Используйте tools (если есть)

**Автоматические инструменты:**

- **Linter (pylint, flake8):** Находит стилистические ошибки
- **Type checker (mypy, pyright):** Проверяет типы
- **Code formatter (black, autopep8):** Форматирует код автоматически
- **Static analysis (bandit):** Находит уязвимости и проблемы безопасности

**Пример:**
```bash
# Проверить код с pylint
pylint your_file.py

# Проверить типы с mypy
mypy your_file.py

# Отформатировать код с black
black your_file.py

# Запустить bandit для проверки безопасности
bandit -r your_file.py
```

**Интеграция с CI/CD:**
```yaml
# .github/workflows/code-quality.yml
name: Code Quality Check

on: [pull_request]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-python@v4
        with:
          python-version: '3.10'
      - run: pip install pylint black bandit
      - run: |
          pylint **/*.py
          black --check **/*.py
          bandit -r **/*.py
```

### 8. Проверяйте тесты (если есть)

**Что проверять в тестах:**

| Аспект | Пример вопроса |
|---------|----------------|
| Покрытие | Тесты покрывают happy path и edge cases? |
| Время выполнения | Тесты выполняются достаточно быстро? |
| Время стабильность | Тесты стабильны и не флакируют (flaky)? |
| Уникальность | Тесты изолированы и независимы друг от друга? |
| Тестовые данные | Используются fixtures и mocks, а не реальные данные? |
| Очистка | Тесты очищают временные файлы? |

**Пример:**
```markdown
В `test_calculate_discount.py` есть только один тест: проверяет правильность расчёта скидки.

Предлагаю добавить следующие тесты:
- `test_zero_discount_rate` — проверяет, что скидка 0% даёт цену без изменения
- `test_negative_discount_rate` — проверяет обработку отрицательных ставок (например, возврат товара)
- `test_full_price_discount` — проверяет, что 100% скидка даёт цену 0
- `test_half_price_discount` — проверяет, что 50% скидка даёт половину цены
```

**Результаты:**
- Более полное покрытие
- Тесты более надёжны
- Меньше вероятность регрессий

### 9. Избегайте "Your vs My" (Ты vs Я)

**Плохо:**
```markdown
Твой код не работает. Мой код лучше.
```

**Хорошо:**
```markdown
Предлагаю объединить наши подходы:

```python
def calculate_total_v1(your_code):
    """Ваша версия функции"""
    pass

def calculate_total_v2(my_code):
    """Моя версия функции"""
    pass

# Затем мы можем сравнить их по производительности
import time

def benchmark_versions():
    """Сравниваем обе версии"""
    start = time.time()
    your_function(test_data)  # 0.1s
    v1_time = time.time() - start
    
    start = time.time()
    my_function(test_data)  # 0.08s
    v2_time = time.time() - start
    
    print(f"Your version: {v1_time:.3f}s")
    print(f"My version: {v2_time:.3f}s")
    print(f"Speedup: {v1_time / v2_time:.1f}x")
```

**Преимущества:**
- Фокус на решении, а не на правоте
- Создаётся дух командной работы
- Могут быть найдены оптимальные решения

### 10. Записывайте решения

**Что документировать:**

- Обнаруженные баги и как они были исправлены
- Производительность и оптимизации
- Архитектурные решения
- Best practices, которые были применены

**Пример:**
```markdown
## Решения от code-review (15.03.2024)

### Bug #1: Негативные цены
**Проблема:** `calculate_total()` не обрабатывала отрицательные цены.

**Решение:** Добавлена валидация входных данных. Цена должна быть положительной.

**Результат:** Функция теперь корректно обрабатывает возвраты товаров с отрицательной ценой.

### Оптимизация #2: Кеширование
**Проблема:** `calculate_total()` вызывался много раз для одних и тех же данных.

**Решение:** Добавлена функция `calculate_total_with_cache()` для кеширования результатов в Redis.

**Результат:** Производительность увеличена на 70%.

### Best Practice: List Comprehensions
**Описание:** Использование list comprehensions вместо циклов for.

**Пример:**
```python
# Вместо:
total = 0
for item in items:
    total += item['price']

# Использовать:
total = sum(item['price'] for item in items if item.get('price', 0) > 0)
```

**Преимущества:** Более Pythonic, читаемо, часто быстрее.

## Quick Reference

| Проверка | Рекомендация | Пример |
|---------|-------------|--------|
| Логика | Проверяйте граничные случаи | Пустой список, None, 0 |
| Производительность | O(n²) → O(n log n) | Не используйте nested loops |
| Читаемость | Следуйте PEP 8 | snake_case, 4-space отступ |
| Тестируемость | TDD, AAA | Сначала пишите тест, потом код |
| Безопасность | Используйте bandit | Проверяйте на уязвимости |
| Документация | Docstrings + Comments | Объясняйте сложные части |
